Java类加载详解

发表于 2020-01-01 | 分类于 Java

1. Java类加载的时机

类加载器（ClassLoader）的具体作用就是将class文件加载到JVM虚拟机中去，保证程序的正确运行。但是，JVM启动的时候，并不会一次性加载所有的class文件，而是根据需要去动态加载。

触发java类初始化的情况有以下几种：

  1. 遇到new（用new实例对象），getStatic（读取一个静态字段，被final修饰的静态常量除外），putstatic（设置一个静态字段，被final修饰的静态常量除外），invokeStatic（调用一个类的静态方法）这4条指令字节码命令时，如果类没有进行过初始化，则会触发类初始化；（被final修饰的静态字段已在编译期存放在常量池）
  2. 使用Java.lang.reflect包的方法对类进行反射调用时，如果此时类没有进行初始化，会先进行初始化；
  3. 当初始化一个类时，如果其父类没有进行初始化，先初始化父类；
  4. JVM启动时，用户需要指定一个执行的主类（包含main的类）虚拟机会先执行这个类。

2. Java内置的类加载器

JDK内置以下3个类加载器:

  • 启动类加载器（Bootstrap ClassLoader）
    最顶层的加载类，主要加载核心类库，%JRE_HOME%\lib下的rt.jar、resources.jar、charsets.jar和class文件等。

  • 扩展类加载器（ExtClassLoader）
    加载目录%JRE_HOME%\lib\ext目录下的jar包和class文件。

  • 应用类加载器（AppClassLoader）
    加载当前应用classpath的所有类。

加载器的加载顺序为：Bootstrap CLassloder -> ExtClassLoader -> AppClassLoader

  1. 由外部类创建ClassLoader时直接指定一个ClassLoader为parent。

Launcher类简单明了地阐述了ExtClassLoader与AppClassLoader的parent来源。

ExtClassLoader并没有直接对parent的赋值，它调用了父类URLClassLoder的构造方法并将parent参数传递为null。

综上所述，AppClassLoader的parent是ExtClassLoader，ExtClassLoader的parent是null，但是Bootstrap CLassLoader却可以当成它的父加载器，这是因为Bootstrap ClassLoader是由C/C++编写的，它本身是虚拟机的一部分，所以它并不是一个Java类，也就是无法在java代码中获取它的引用。JVM启动时通过Bootstrap类加载器加载rt.jar等核心jar包中的class文件，int.class、String.class等都是由它加载。然后JVM初始化sun.misc.Launcher并创建Extension ClassLoader和AppClassLoader实例，并将ExtClassLoader设置为AppClassLoader的父加载器，Bootstrap没有父加载器，但是它却可以作用一个ClassLoader的父加载器。所以在双亲委派的机制下，当类加载请求递归至ExtClassLoader时，ExtClassLoader会将加载请求传递至Bootstrap CLassLoader。

3. 双亲委派

双亲委派的工作原理是，如果一个类加载器收到了类加载请求，它首先判断这个class是否已经加载成功，如果没有加载成功并不会自己先去加载，而是把这个请求委托给父加载器去执行，如果父类加载器也存在父类加载器，则进一步向上委托，依次递归，请求最终将到达顶层的启动类加载器Bootstrap CLassLoader；如果父类加载器可以完成类加载任务，就成功返回，倘若父类加载器无法完成此加载任务，子加载器才会尝试自己去加载，这就是双亲委派模式。

具体步骤如下：

  1. 一个AppClassLoader查找资源时，先看看缓存是否有，缓存有从缓存中获取，否则委托给父加载器ExtClassLoader，如果ExtClassLoader也没有加载过，则递归向上由Bootstrap ClassLoader查找。
  2. Bootstrap ClassLoader首先查找缓存，如果没有找到的话，就去找自己的规定的路径下，也就是sun.mic.boot.class下面的路径，找到就返回，没有找到则让子加载器自己查找。
  3. Bootstrap ClassLoader如果没有查找成功，则ExtClassLoader自己在java.ext.dirs路径中去查找，查找成功就返回，查找不成功则向下让子加载器查找。
  4. ExtClassLoader查找不成功，AppClassLoader就自己查找，在java.class.path路径下查找。找到就返回。如果没有找到就让子类找，如果没有子类会抛出异常。

从双亲委派的加载流程可以发现，委托是从下向上，具体的查找过程却是从上至下。

4. 源码分析

4.1 loadClass()

判断类是否已加载，使用双亲委派模型，请求父加载器，如果加载结果都为空，调用findclass()。

方法的具体逻辑如下：

  1. 执行findLoadedClass(String)判断类是否已经加载过；
  2. 执行父加载器的loadClass()方法。如果父加载器为null，则使用JVM内置的类加载器Bootstrap ClassLoader去替代。这也表明，即便ExtClassLoader的parent为null，但是可以将Bootstrap ClassLoader理解为它的父加载器；
  3. 如果向上委托父加载器没有加载成功，则通过findClass(String)查找；
  4. 如果类在上面的步骤中找到了，且参数resolve为true，则loadClass()会调用resolveClass(Class)这个方法来生成最终的Class对象。

代码清晰地解释了双亲委派模型的类加载逻辑。如果需要自定义一个classloader，建议覆盖findClass()方法，而不要直接改写loadClass()方法，修改双亲委派的模型。

4.2 findclass()

根据名称或位置加载.class字节码，然后调用defineClass()进行解析。

4.3 defineClass()

解析.class字节流，返回Class对象。一般建议在自定义ClassLoader的findclass()方法中调用基类的该方法，将class二进制内容转换成Class对象。

5. 自定义ClassLoader

Java内置的类加载器（Bootstrap ClassLoader、ExtClassLoader等），只能加载指定的目录下的jar包或者资源。如果需要从磁盘、内存或网络等来源动态加载class文件时，则需要自定义一个类加载器。自定义类加载器的步骤如下：

  1. 创建一个继承ClassLoader的类。
  2. 复写抽象父类的findClass()方法。
  3. 在findClass()方法中调用defineClass()。

注意：父类有那么多方法，为什么只复写findClass()方法？

因为JDK已经在loadClass()方法中帮我们实现了ClassLoader搜索类的算法，当父类的loadClass()方法中加载不到目标类时，就会调用本类的findClass()方法来搜索类，所以只需重写该方法即可。如没有特殊的要求，一般不建议重写loadClass()加载类的算法。
