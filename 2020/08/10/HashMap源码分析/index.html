HashMap源码分析

发表于 2020-08-10 | 分类于 Java

1. HashMap的定义和结构

1.1 HashMap的概念

HashMap是一个用于存储Key-Value键值对的数据结构，每一个键值对节点用Node<K,V>表示

Node是一个内部类，key为键，value为值，next指向下一个元素，即HashMap中的元素不是一个单纯的键值对，还包含key的hash值和下一个元素的引用。

1.2 HashMap的特性

HashMap具有如下特性：

  • HashMap 是一个散列表，它存储的内容是键值对(key-value)映射
  • HashMap的实现不是同步的，即它不是线程安全的。注意与Hashtable、ConcurrentHashMap的区别
  • HashMap的key和value都可以为null。注意与TreeMap、ConcurrentHashMap的区别
  • HashMap中的映射不是有序的
  • HashMap有两个参数影响其性能：初始容量和加载因子。默认初始容量是16，加载因子是0.75。容量是哈希表中桶(Entry数组)的数量，初始容量只是哈希表在创建时的容量。加载因子是哈希表其容量自动增加的阈值（比例），当哈希表中的元素个数超出了加载因子与当前容量的乘积时，通过调用 rehash 方法将容量翻倍。

2.6 HashMap.put的源码分析

put函数的具体执行过程如下：

  1. 判断哈希数组是否为空，如果是空，则调用resize()方法创建默认长度位 16 的数组；
  2. 通过与运算计算对应 hash 值的下标，如果对应下标的位置没有元素，则直接创建一个单链表节点赋给数组中索引下的这个节点；
  3. 如果有元素，说明 hash 冲突了，则进行 3 种分支判断：
    • 3.1 使用 equals 方法判断两个冲突的key是否相等，如果相等，则将已经存在的值赋给变量e，并更新e的value；
    • 3.2 如果key不相等，则判断节点是否红黑树类型，如果是红黑树，则交给红黑树追加此元素；
    • 3.3 如果key既不相等，节点也不是红黑树类型，则说明节点是链表元素，此时遍历链表中的每一个key和给定的key是否相等。如果没找到key对应的元素，且链表的长度大于等于8，则将链表改为红黑树。
  4. 最后，如果步骤3中的三个判断返回的 e 不为null，则说明key重复，更新key对应的value的值，并返回旧值。
  5. 对维护着迭代器的 modCount 变量加1。
  6. 最后，如果HashMap包含的元素个数已经大于阀值了，则对数组进行扩容，并重新散列元素。

2.7 HashMap.get的源码分析

get函数的具体执行过程如下：

  1. 如果数组不为null，且数组的长度大于0，且指定的索引值（元素key对应的hash值）不为null，则进一步判断，否则直接返回null；
  2. 如果数组下标的首个元素匹配成功（元素的hash值和key值均等于查询元素的对应值），则返回该元素；
  3. 若首元素节点匹配不成功，首节点的next节点不为null，则：
    • 3.1 若首节点的类型为红黑树的TreeNode，则通过红黑树查找并返回匹配值；
    • 3.2 若首节点的类型不是红黑树节点，则next节点为单向链表节点，通过do while循环查询给定值，匹配成功则返回。

2.8 HashMap.resize(扩容)的源码分析

函数的具体执行过程可分为两部分：扩容数组 和 重新散列元素。

扩容数组的具体过程如下：

  1. 判定数组容量是否已达到最大值，如果是则不再扩容，直接将老的数组返回；若新数组大小（老数组容量* 2）小于数组最大值（int最大值），并且老数组容量大于等于数组初始化大小（16），则更新阈值（老阈值 * 2）；
  2. 如果老的阀值大于0，则新的容量等于老的阀值；
  3. 如果老的阀值和容量都不大于0，则容量设置为默认初始容量为16，阀值为 16 * 0.75f，也就是 12；
  4. 如果新的阀值还是0，那么就使用刚设置的容量乘以0.75作为新的阀值，并判断算出的阀值是否合法：如果容量小于最大容量并且阀值小于最大容量，那么则使用该阀值，否则使用int最大值；
  5. 将经过上述步骤计算好的阈值和数组设置为HashMap的属性值。

重新散列元素的具体过程如下：

  1. 首先循环老数组。下标从0开始，如果对应下标的值不是null，则判断该值有没有next节点，即是否链表；
  2. 如果不是链表，则根据新的数组长度重新hash该元素；
  3. 如果该节点是树，则调用红黑树的 split 方法拆分红黑树，传入当前对象和新数组还有下标，该方法会重新计算红黑树中的每个hash值，如果重新计算后，树中元素的hash值不同，则重新散列各元素到不同的数组下标中；
  4. 否则数组下标的元素就是链表，此时对旧的链表元素进行重新 hash，如果 hash 得到的值和之前不同，则从旧的链表中删除，放到新的数组下标中去。

3.1 HashMap的线程安全性分析

3.1.1 HashMap的线程安全性问题

(1) 多线程put时可能会导致get无限循环，具体表现为CPU使用率100%；
原因：在向HashMap中put元素时，会检查HashMap的容量是否已经达到阈值，如果达到阈值，则会新建一个两倍原容量大小的Hash表，然后把元素重新hash到新表中，多个线程同时操作就有可能会形成循环链表，此时使用get()时，可能出现无线循环的情况。

(2) 多线程put时可能导致元素丢失
原因：当多个线程同时执行addEntry(hash, key, value, i)时，如果产生哈希碰撞，导致两个线程得到同样的bucketIndex去存储数据，就可能会发生元素覆盖丢失的情况。

综上可知，HashMap是线程不安全的。

3.1.2 解决HashMap线程安全性问题的方法

(1) 使用Hashtable替代HashMap

(2) 使用Collections.synchronizedMap封装HashMap
synchronizedMap封装的原理是把HashMap赋值给Collections的成员变量，通过synchronized关键字将get和put等方法加上同步机制。优点是使用比较简单，缺点是性能较差。

(3) 使用ConcurrentHashMap替代HashMap
